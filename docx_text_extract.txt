Vistas Nuevas                   Convención para Nombres                     El nombre de la vista debe comenzar con el          prefijo del proyecto          seguido de         l código del módulo,          una descripción y colocando al final el sufijo “_V”. Si se trata de una vista multi-org, el sufijo “_V” no debe estar presente. Para las vistas materializadas el sufijo debe ser “_MV” El formato es el siguiente:                    &lt;Proyecto&gt;         _&lt;Módulo&gt;_&lt;Descripción&gt;                    XXFC_&lt;Módulo&gt;_&lt;Descripción&gt;[_V/_MV]                               Las vistas multi-org, son aquellas basadas en tablas _ALL que se encargan de filtrar el ORG_ID que corresponda. El nombre de la vista debe coincidir con el nombre de la tabla multi-          org           pero sin el _ALL (y sin colocarle el prefijo _V).          Se sugiere que l         a longitud máxima del nombre para este tipo de vistas es 23 caracteres. Por ejemplo, si la tabla multi-org se llama XXFC_AR_PROFILES_ALL la vista entonces debe llamarse XXFC_AR_PROFILES.                               Las vistas join y vistas materializadas son aquellas que involucran a más de una tabla en su definición. El nombre de estas vistas debería ser una combinación de los objetos involucrados agregándole el sufijo _V ó _MV.          Se sugiere que l         a longitud máxima para el nombre en este caso es de 25 caracteres. Por          ejemplo,          si las tablas son: XXFC_OM_ORDERS y XXFC_OM_LINES y se quiere realizar una vista que muestre datos de ambas, el nombre de la misma podría ser (entre otros) XXFC_OM_ORDER_LINES_V. Un código fuente que se vea bien en un editor, puede no verse tan bien en otro. Esto se debe al distinto manejo del carácter “tab” que cada editor haga. Bloqueos y Waits                      En Worfkflow existen dos maneras de frenar un proceso automáticamente (sin intervención del usuario): Blocks y Waits. Otra opción es implementarlo con Timeout, haciendo un ciclo para que vuelva a una actividad dependiendo de alguna condición.                Bloqueos                      Se puede forzar que el flujo se suspenda utilizando una acitivad de Bloqueo. Internamente esto es similar a una notificación cuyo estado esta en NOTIFIED.Una vez que un proceso queda bloqueado, no continúa hasta que se invoque una API del Workflow (CompleteActivity) que se encarga de desbloquearlo y luego el flujo continúa normalmente.                     Esta solución es performante pero a veces poco viable ya que la condición a cumplir para desbloquear es dificil de detectar automáticamente (vía triggers generalmente). Se utiliza para sincronizar con algún evento externo identificable.                Waits                      De utilizarse un Wait, el proceso del workflow no continúa hasta que pasa el tiempo fijado por la actividad de Wait. Esta operación de chequeo del tiempo transcurrido la realiza el concurrente “Workflow Background Process” siempre que se lance con el parámetro Deferred en Yes.                      Los waits presentan una opción con impacto en la performance pero son mas viables de hacer ya que se pueden implementar con un package que valide la condición a alcanzar en la base de datos.                     Al parametrizar el wait, se debe definir el attributo “Wait Mode” a nivel de actividad que define si el modo de espera es relativo o absoluto. Si se elige relativo, hay que poner un valor en el atributo “Relative Time” con el siguiente formato: dias.fraccion_dias. Por ejemplo, si se quiere tener un wait que se active cada 5 minutos, hay que poner: 0.0034. Esto sale de: 0 dias, y el 0034 de la cuenta 5 / 1440 (minutos en 1 dia)          . Utiliza principalmente Java y el patrón de diseño Model-View-Controller.                              La vista está implementada con XMLs que podemos manejar como metadata desde JDeveloper. OAF transforma los XMLs en HTML en tiempo de ejecución.                              El controlador es implementado por clases Java similares a Servlets que enrutan la lógica según las acciones del usuario. Si quieres cambiar el aspecto de la pantalla entre requests, se hace aquí.                              El modelo está implementado en varias capas. Primero los llamados “ApplicationModules”, que encapsulan las transacciones y llaman a las capas de acceso a datos:                              View Objects                              A grandes rasgos, un View Object representa un query a la base de datos. Puede ser cualquier query que vayamos a utilizar para desplegar los resultados en pantalla (como tablas, etc.)                              Puede estar conformado por un query escrito por nosotros, o por un query generado automáticamente en base a EntityObjects.                              Entity Objects                              Archivos XML que contienen información de las tablas de la base de datos, y su lógica de validación. Cuando queremos guardar información en tablas, los “sets” de estas clases hacen las validaciones necesarias.                Flujo principal de una página OAF                   En una pantalla OAF, un flujo por lo regular sigue los siguientes pasos:                       El cliente inicia un request, dando clic sobre un           link         ,           submitButton         , etc.                       Se llama al Controller que corresponde al botón o al padre más cercano (asociado desde el xml).                       Si es la primera vez que entramos a una página, primero se llama al método           processRequest          en el controller.                       En el Controller se ejecuta el método           processFormRequest         .                        Del Controller (dependiendo de la lógica) se pueden llamar varios métodos del ApplicationModule asociado a la página.                       El ApplicationModule es el que mantiene la transacción del usuario. Se encarga de ejecutar queries (con procedures o con ViewObjects) y hacer commits y rollbacks a la base de datos.                       Como ejemplo, el AM ejecuta el query de un ViewObject.                       Se regresa al Controller y por fin al usuario. Si hay una tabla asociada al ViewObject, se actualiza su información en pantalla.                              17780    34925                                                                          Componentes comunes de OAF                  Tablas                   En          OAF existen dos tipos de tablas:                       Table                       AdvancedTable                           Su funcionamiento es muy similar, pero ‘AdvancedTable’ provee más funcionalidad que Table. Por ejemplo, AdvancedTable permite headers que agrupen otros headers, y permite que dentro de las celdas haya otros componentes más complicados que los que permite ‘Table’.                           El propósito de las tablas es principalmente el despliegue y modificación de información. Con OAF se facilitan mucho ambos escenarios.                           Ésta funcionalidad la obtienen asociándose a una           instancia de un ViewObject         . Ya que un ViewObject puede decirse que representa una consulta a la base de datos, pueden crearse varias instancias de un ViewObject. Por ejemplo:                       Tenemos un ViewObject llamado PersonaVO, el cual contiene una consulta para obtener el catálogo de personas.                       De ese ViewObject, le tenemos que asignar una instancia al ApplicationModule (ya que los ViewObjects pueden utilizarse en muchos lugares con distintos ApplicationModules).                       Entonces a nuestro ApplicationModule (AdminPersonasAM) le asignamos una instancia, que por defecto se llamará ‘PersonaVO1’.                       Nosotros podemos llamar métodos sobre esa instancia, por ejemplo ‘          executeQuery(          )’ para hacer la consulta inicial a la base de datos.                           OAF maneja por defecto la paginación de resultados en las tablas. Al dar clic en “Siguiente” ó “Anterior”, se realiza un request y OAF actualiza los registros en el VO haciendo un nuevo query. Como la información del VO cambió y la tabla está           asociada          al VO, al regresar la respuesta al browser se verán los nuevos registros. Codificación Java                       Los estándares de codificación están basados en los recomendados por ORACLE-Sun, que han sido difundidos y aceptados ampliamente por toda la comunidad Java, y que han terminado por consolidarse como un modelo estándar de programación.                    Estas normas son muy útiles por muchas razones, entre las que destacan:                         Facilitan el mantenimiento de una aplicación. Dicho mantenimiento constituye el 80% del coste del ciclo de vida de la aplicación.                         Permite que cualquier programador entienda y pueda mantener la aplicación. En muy raras ocasiones una misma aplicación es mantenida por su autor original.                         Los estándares de programación mejoran la legibilidad del código, al mismo tiempo que permiten su compresión rápida.                       Organización de Archivos Fuente                    Las clases en Java se agrupan en paquetes. Estos paquetes se deben organizar de manera jerárquica, de forma que todo código desarrollado para FEMSA Comercio deben de iniciar sin excepción            “            com.femsa            .oxxo”          .                  Dentro del paquete principal las clases se organizarán en subpaquetes en función del nombre del proyecto clave, al que pertenezca el código desarrollado.                   Por ejemplo, si estamos desarrollando un proyecto web de evaluaciones cuyo nombre clave del proyecto es SISADE, las clases de dicho proyecto se incluirían en el paquete            "com.femsa.oxxo.sisade"          o similar.                  Un archivo de código fuente consta de secciones que deben estar separadas por líneas en blanco y comentarios opcionales que identifiquen cada sección.                  Deben evitarse los archivos de gran tamaño que contengan más de           1000          líneas. En ocasiones, este tamaño excesivo provoca que la clase no encapsule un comportamiento claramente definido, albergando una gran cantidad de métodos que realizan tareas funcional o conceptualmente heterogéneas.                  Cada archivo fuente debe contener una única clase o interfaz pública.                   El nombre del archivo tiene que coincidir con el nombre de la clase.                   Cuando existan varias clases privadas asociadas funcionalmente a una clase pública, podrán colocarse en el mismo archivo fuente que la clase pública.                   La clase pública debe estar situada en primer lugar dentro del archivo fuente.                  En todo archivo fuente distinguimos las siguientes secciones:                        Comentarios de inicio.                        Sentencia de paquete.                         Sentencias de importación.                        Declaraciones de clases e interfaces. Versión 3.3                         Se debe ejecutar dentro de un ciclo de parchado, se instala en ambiente "PATCH".           * Debe dejar el driver de sincronización preparado para que los cambios hechos en PATCH se repliquen al iniciar el siguiente ciclo de parchado.           * Los cambios no se verán reflejados hasta cerrar el ciclo de parchado (           cutover          )                         Debe haber un ciclo de parcheo abierto, ya que contiene componentes de aplicación que se instalarán sobre el            file system           de PATCH. Seguridad por Datos                     Control de acceso sobre datos que el usuario puede acceder. Se restringe el acceso a datos individuales que son desplegados en pantalla una vez que el us          uario ha seleccionado una funció          n del menu.                  Control Basado en Roles                     Los accesos de usuario al sistema son determinados por roles que le son otorgados. Un rol puede ser configurado para consolidar responsabilidades, permisos, politicas de funciones y datos de seguridad que el usuario requiere pare ejecutar una funcion especifica.                Delegando Administración                     Con la delegación de administración, se pueden crear administrsadores locales y otorgarles los permisos suficientes para administrar un conjunto específico de usuarios y roles dentro de la organización.                    Dando Servicios                                Habilita              a usuarios finales a ejecutar algunas de sus propias tareas de registro, como por ejemplo: nuevas cuentas o accesos adicionales al sistema.                Auto Servicio &amp; Aprobaciones                                 Tareas subsecuentes al registro como por ejemplo solicitud de nuevos accesos al sistema.                        Oracle User Management – Configuración                                 Se configura el control de acceso y políticas de seguridad definiendo roles, jerarquías de roles, categorías de roles y procesos de registro.                         Seguridad  de            Oracle Application Libray                                 Se configuran usuarios y se asignan una o más responsabilidades.
